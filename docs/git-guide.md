# E.O.M Git Guide

## 1. 브랜치 전략

```text
main  <- 배포 가능한 안정 버전
dev   <- 통합 개발 브랜치
feat/*   기능 개발
fix/*    버그 수정
docs/*   문서 작업
design/* 디자인/화면 작업
chore/*  설정/정리 작업
```

흐름:

```text
feat/* -> PR -> dev -> PR -> main
```

## 2. main 규칙

- `main`은 항상 배포 가능한 상태로 유지합니다.
- 설계서, PRD, 로드맵 같은 `docs/` 문서는 `main`에 올리지 않습니다.
- 루트 `README.md`는 프로젝트 설명이므로 `main`에 유지합니다.

## 3. dev 규칙

- 문서와 구현을 통합하는 브랜치입니다.
- 실제 기능 구현은 가능하면 하위 브랜치에서 진행합니다.
- 여러 작업이 섞이지 않도록 작은 PR 단위로 합칩니다.

## 4. 커밋 컨벤션

형식:

```text
type: 한국어 요약
```

type:
- `feat`: 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 추가/수정
- `design`: UI, 스타일, 테마
- `refactor`: 동작 변화 없는 코드 정리
- `chore`: 빌드, 설정, 의존성, 정리
- `test`: 테스트 추가/수정

예시:

```text
feat: 로그인 기본 구조 추가
feat(board): SHOW 게시판 목록 구현
fix: 관리자 페이지 권한 체크 수정
docs: PRD 요구사항 정리
design: 대시보드 다크 테마 적용
chore: Gradle 설정 정리
test: 로그인 흐름 테스트 추가
```

## 5. PR 컨벤션

타겟:
- 기능/문서/디자인 작업은 `dev`로 PR
- 릴리스 준비 완료 후 `dev`에서 `main`으로 PR

제목:

```text
[type] 한 줄 요약
```

본문:

```markdown
## 작업 내용
- 핵심 변경 사항

## 확인 방법
- 실행/테스트 방법

## 체크리스트
- [ ] 로컬 실행 확인
- [ ] 관련 화면 확인
- [ ] 문서 업데이트
```

## 6. 작업 루틴

1. `dev` 최신화
2. 하위 브랜치 생성
3. 작은 단위로 구현
4. 테스트 또는 화면 확인
5. 관련 문서 업데이트
6. 커밋
7. `dev`로 PR

## 7. 커밋 크기 원칙

- 한 커밋은 한 가지 목적만 담습니다.
- 기능과 문서가 강하게 연결되면 같은 PR에 두되, 커밋은 분리할 수 있습니다.
- 마감 중에도 너무 큰 커밋 하나로 몰지 않습니다.
